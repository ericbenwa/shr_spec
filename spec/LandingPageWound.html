This is a preliminary version of MITRE's "Mini-CIMI" version of Skin and Wound FHIR Implementation Guide, based on Standard Health Record (SHR) base classes, including FHIR logical models, FHIR profiles, value sets, and extensions.

CIMI Skin and Wound Demonstration Project

The Clinical Information Modeling Initiative (CIMI) is focused on creating logical models of healthcare information, independent of implementation technologies and patterns. It aims to decouple the logical information content from the periodic changes we experience in technology. This IG is part of a demonstration project designed to be a test case for how different groups, with different tools, approach CIMI modeling. Each group is using the same Excel spreadsheet describing the Skin and Wound domain.

Background: Why Is "Mini-CIMI" Needed?

CIMI has established an object-oriented hierarchy that defines over 200 classes. Creating CIMI Domain-specific models involves four different modeling processes:

  1. Selecting a CIMI-defined base class or base archetype (required);
  2. Extending the base class (adding new attributes);
  3. Constraining the derived child class or archetype (by restricting cardinality, applying fixed values to attributes, binding properties to value sets, or constraining the type of attributes);
  4. If required, combining several classes through composition to form the desired domain model (for example, choosing the correct StatementTopic and StatementContext classes to form the desired type of clinical statement).

Further, the modeling actions involve to two different representations: BMM for extending the class hierarchy and composing classes, and ADL for applying constraints. Even though tooling can in theory hide this gap, in practice, there are still no user-friendly tools that accomplish that. 

The current author (Mark Kramer) has significant experience with CIMI modeling, having personally produced several FHIR implementation guides using CIMI modeling, including the HL7 FHIR® Implementation Guide: Breast Cancer Data, Release 1 - US Realm and the HL7 FHIR Profile: Occupational Data for Health (ODH), Release 1 (Standard for Trial Use). The author has created a complete toolchain and domain-specific language for creating CIMI models, regularly participated in the CIMI workgroup, and made literally hundreds of contributions to the CIMI reference model.

Orchestrating subclassing, constraining, and composing involves literally hundreds of modeling decisions, such as:

  • Which CIMI classes or archetypes should be used to represent clinical concepts? For example, what hair-splitting differences between assertions, findings, evaluations, assessments. 
  • Which part or parts of the CIMI hierarchy should be extended, ClinicalStatement, StatementTopic, or StatementContext? 
  • What attributes belong in StatementTopic, and which in StatementContext?
  • At which point in the inheritance tree should constraints be added?
  • How to manipulate (or hide) various features of the CIMI reference model to achieve reasonable mapping to FHIR?
  • How to achieve domain modularity (separating breast cancer profiles, for example) into self-contained groups of profiles, when all models in the CIMI logical model are connected?
